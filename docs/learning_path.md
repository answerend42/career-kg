# 成长路径编排

## 目标

成长路径层解决的问题不是“最适合你做什么”，而是“如果你指定想做某个岗位，接下来 2 到 4 步先补什么、为什么这样排”。

核心约束仍然是知识图谱驱动：

- 步骤来源于图中的父节点缺口
- 顺序受 `requires / supports` 和当前门槛状态影响
- 每一步都要能落回具体节点和 boost 证据

## 输入

成长路径不单独开放新接口，而是挂在 `/api/role-gap` 的 `target_role.learning_path` 中。

规划器复用：

- 用户归一化后的 `score_map`
- 图推理后的 `states`
- `RoleGapAnalyzer.build_gap_suggestions(...)`
- 建议节点回溯出的证据 boost

## 编排规则

1. 从目标岗位当前缺口中取 4 到 6 个候选建议节点。
2. 对每个候选建议节点，向上游追溯可补的证据节点。
3. 用轻量注入重算该单步动作对目标岗位的预估增益。
4. 对候选动作排序：
   - `requires` 优先于 `supports`
   - 单步增益更高者优先
   - 能带动更多中间节点者优先
   - 已经重复使用过的 focus 节点会被惩罚
5. 选出当前最优动作后，把该步 boost 写回临时分值，再规划下一步。
6. 如果没有正增益动作，或目标岗位分数已进入“基本就绪”区间，则提前停止。

## 为什么要复用证据 boost

图里很多能力节点是 `soft_and` 聚合。  
如果直接把中间能力节点抬高，未必会让目标岗位真的涨分。

因此成长路径里的每一步也复用了 what-if 的证据 boost 逻辑：

- 动作是“补某个能力/方向”
- 真正注入的是该能力背后的项目、知识或工具证据节点

这样用户看到的步骤既有抽象目标，也有可执行的落点。

## 行动模板层

在当前实现里，每个成长路径步骤还会继续挂一层 `recommended_actions`：

- 优先返回项目型动作
- 再补练习型或作品集型动作
- 匹配依据不是自由生成文案，而是模板库与 `focus_node_id / boosts / target_role_id` 的覆盖关系

这让系统可以从“补哪些节点”继续落到“具体做哪类项目、练习或交付物”。

在此基础上，当前还增加了 action simulation：

- 用户选中某个 `recommended_actions[*]`
- 后端读取其中的 `simulation_node_ids`
- 对这些证据节点做轻量注入
- 再次运行整张图的传播，返回目标岗位和岗位排序的变化

## 返回字段

每个 `learning_path[*]` 包含：

- `step`
  - 第几步
- `focus_node_id` / `focus_node_name`
  - 这一阶段主要围绕哪个图节点补强
- `relation`
  - 当前步骤对应的是 `requires` 还是 `supports`
- `title`
  - 前端可直接展示的步骤标题
- `summary`
  - 为什么这一步排在这里、预估能带来什么提升
- `expected_score_delta`
  - 做完这一步后，目标岗位预估增加多少分
- `expected_total_score`
  - 做完这一步后，目标岗位预计达到多少分
- `blocked_by`
  - 这一步主要是在解决哪些门槛/卡点
- `unlock_nodes`
  - 这一步会显著带动哪些中间能力或方向节点
- `boosts`
  - 这一阶段实际补的证据节点
- `recommended_actions`
  - 当前这一步最值得执行的 1 到 2 个行动模板
  - 每个模板还会带 `simulation_node_ids`，供 `/api/action-simulate` 复用

## 当前边界

- 这是稳定、可解释的启发式编排器，不是全局最优规划器。
- 当前模板库仍是本地维护的小规模高质量集合，还不是外部课程/项目资源平台。
- 当前会尽量避免重复 boost，但仍允许同一 focus 节点分阶段推进，以免遗漏真实高杠杆步骤。
