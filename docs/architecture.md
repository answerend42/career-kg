# 架构说明

## 1. 图谱分层

当前图谱固定为 5 层 DAG，并通过 source 数据编译得到运行时 seed：

1. `evidence`
   - 技能、工具、课程、项目、偏好、软技能、约束
2. `ability`
   - 编程基础、数据库实践、后端基础、数据处理、机器学习基础等
3. `composite`
   - 后端工程能力、数据工程能力、机器学习工程能力等
4. `direction`
   - Web 后端、数据、机器学习、运维平台等岗位方向
5. `role`
   - 后端开发工程师、Python 后端工程师、数据工程师等具体岗位

## 2. 边语义

- `supports`: 常规正向支持
- `requires`: 关键前置，参与门槛判断
- `prefers`: 偏好加成
- `inhibits`: 抑制因素
- `evidences`: 项目和课程等实践性证据

## 3. 聚合器

- `source`
  - 直接使用输入分值
- `weighted_sum_capped`
  - 支持大多数基础能力节点
  - 聚合多种证据并做上限裁剪
- `max_pool`
  - 适合“单个强证据即可显著抬升”的节点
- `soft_and`
  - 适合复合能力节点
  - 使用覆盖率控制“只靠单一路径虚高”
- `penalty_gate`
  - 适合方向节点
  - 若关键前置不足，不归零但折减
- `hard_gate`
  - 适合职业节点
  - 关键前置明显缺失时直接归零

## 4. 避免重复计数

推理引擎不会简单把所有父节点线性求和。它会：

- 记录每个节点的根证据贡献映射
- 同一根证据通过多条路径到达同一节点时，采用按根取最大值的策略，降低重复计分
- 在复合能力层引入覆盖率，避免单一兴趣或单一技能把整个方向抬得过高

## 5. 可解释性输出

每次推荐返回：

- `normalized_inputs`
  - 用户输入被映射到哪些标准节点
- `recommendations[*].paths`
  - 从根证据到职业节点的高贡献路径
- `propagation_snapshot`
  - 已激活节点、得分与高贡献边，可直接驱动前端传播可视化

## 6. 数据构建

- `data/sources/skills.json`
  - 原子技能、工具、课程、项目、偏好、约束等证据节点
- `data/sources/capability_templates.json`
  - 基础能力、复合能力、方向模板
- `data/sources/roles.json`
  - 通用岗位和模板化细分岗位定义
- `scripts/build_graph.py`
  - 将 source 数据编译为 `data/seeds/*.json` 和词典文件

这意味着图谱扩容优先在 source 层完成，而不是直接手改 seed。

## 7. 当前限制

- 自然语言解析仍是词典 + 模式匹配，后续可接入 LLM 做候选实体抽取，但推荐核心仍应保留在图推理层
- 当前 source 数据仍以课程项目演示规模为主，后续可继续接入爬取数据或外部职位画像
- 前端工作台已经接入，当前限制更多在图谱继续扩容和自然语言抽取精度
