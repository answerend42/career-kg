## Trace 1: External Profile To Provenance

Sample profile:

- `profile_id`: `onet_data_scientists`
- `source_type`: `onet_online`
- `source_url`: `https://www.onetonline.org/link/summary/15-2051.00`
- `mapped_node_ids`: `cap_ml_engineering`, `cap_data_analysis`, `dir_machine_learning`, `dir_data_analytics`, `role_ml_engineer`

Current flow:

1. `data/sources/raw/onet_manifest.json` declares the external profile and mapped internal nodes.
2. `scripts/import_external_profiles.py` fetches/parses the source and writes `data/sources/imported_profiles.json`.
3. `scripts/build_graph.py` merges matching profile data into runtime node metadata.
4. `data/seeds/nodes.json` stores `metadata.source_refs` on mapped nodes.
5. API responses expose provenance via recommendation `source_refs`.

Observed limitation: this is still mostly source anchoring plus manual `mapped_node_ids`; it is not yet a full raw -> mention -> canonical concept -> triple -> serving graph pipeline.

## Trace 3: Recommendation To Path Explanation

For the same sample text, the top recommendation is:

- `role_data_engineer`
- Name: 数据工程师
- Score: `0.1401`

First explanation path:

```text
偏好数据 -> 数据工程师
relation: prefers
path score: 0.1032
```

Current flow:

1. Normalized evidence scores enter `InferenceEngine`.
2. Scores propagate over `data/seeds/edges.json`.
3. `RecommendationService` ranks role nodes above the minimum threshold.
4. `GraphExplainer` returns top contribution paths.
5. Frontend consumes `recommendations[*].paths` for graph/path explanation.

Observed limitation: the first path can be very short when direct preference edges dominate. The canonical KG work should make derivation metadata and source support more explicit, so explanations can show why a serving edge exists.
